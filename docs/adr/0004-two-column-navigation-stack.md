# ADR-0004: Two-Column Navigation Stack

## Status

Accepted

## Context

До этого решения приложение имело жёстко захардкоженную структуру: `App` → `MRSplitView` (left: MR list, right: MR detail). Добавление новых фич (Issues, Pipelines) требовало бы дублирования этой структуры. UI не имел единой точки входа — каждый экран сам управлял разбиением на панели.

Нужна расширяемая структура с консистентным UI и понятными переходами.

## Decision

Ввести Navigator с Navigation Stack и двухколоночным layout'ом:

1. **Navigation Stack** — массив `Screen[]`. `push()` / `pop()` = переходы. Рендер = полная перерисовка (без анимации).

2. **Screen** — `{ id, component, props }`. Компонент сам рендерит Left + Right Panel, координируя их через локальный стейт и коллбэки (`onChangeFile` и пр.).

3. **Колонки**: Left = контекст уровня N-1 (без фокуса, реактивный), Right = активный уровень N. Ширина: 30/70 по умолчанию, переопределяется через `Config.ui.leftColumnWidth`.

4. **NavigationContext**: `useNavigation()` возвращает `{ push, pop, setHints }`. Доступен любому компоненту внутри Screen без прокидывания пропов.

5. **Status Bar**: глобальный, двухстрочный (контекстные хоткеи + глобальные). Управляется через `setHints()`.

## Alternatives considered

- **Единый компонент с mode prop** — Screen получает `mode: 'left' | 'right'` и сам адаптируется. Отклонено: узкая левая панель ломает компоненты, оптимизированные под широкий right panel.
- **Левая панель как предыдущий Screen с `focused=false`** — проще, но размазывает компоненты на узкой панели.
- **Пропы вместо контекста для навигации** — явнее, но прокидывать через 3+ уровня компонентов неудобно.

## Consequences

- Каждый новый Screen реализует два layout'а: левый (компактный) и правый (полный).
- Deep links могут восстанавливать стек по `id` экранов.
- `MRSplitView` рефакторится в отдельные Screen'ы: `MRListScreen`, `MRDetailScreen`, `DiffScreen`.
